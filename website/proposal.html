AEGIS-172: Cloud-Adaptive Two-Player Radar Defense Arena

A standalone embedded game system where one player defends sectors and one player applies pressure through cloud-driven threat waves. The platform combines deterministic real-time control with AWS mission generation, scoring, and replay artifacts.

1. Description

AEGIS-172 is an interactive tabletop defense arena. The device scans the environment with a stepper-driven ultrasonic radar and identifies threat sectors using fused signals from ultrasonic distance, ambient light change, tilt events, and thermal context. The defender controls a servo shield and tactical actions through a joystick and IR remote while a second player influences round pressure through mode selection and timing strategy.

Each match is cloud-adaptive. The device requests a mission profile from AWS IoT, Lambda generates wave scripts, and S3 stores mission files and replay artifacts. At run completion, Lambda computes score and writes leaderboard data, allowing reproducible verification of round behavior and objective grading evidence.

Similar class projects often focus on locks, simple games, or one-sensor automation. AEGIS-172 targets higher complexity with coupled mechatronics, sensor fusion, cloud-generated mission logic, and two-player interaction in one standalone system.

2. Design

2.1 Functional Specification

The match engine is organized as a strict state machine with cloud participation in the critical path and local fallback mission logic for resilience.

BOOT + SELFTEST

Initialize displays, actuators, and communication channels; verify sensor sanity.

CALIBRATE

Capture baseline noise floors for ultrasonic, light, and tilt; lock baseline hash.

MISSION FETCH

Request mission from AWS IoT; consume Lambda-generated mission file from S3.

ACTIVE ROUND

Threat scan, defender actions, scoring stream, and on-device visual/audio feedback.

CLOUD JUDGE

Lambda computes score adjustments, winner, and next-round wave seed.

FAILSAFE / LOCAL MODE

If cloud unavailable, run deterministic local mission profile and queue telemetry.

ROUND END

Show final score on LCD/7-segment and publish signed run summary.

REPLAY READY

Persist replay artifact to S3 for demonstration proof and reproducibility.

2.2 System Architecture

CC3200 Main Controller

  • Owns global state machine and round orchestration
  • Handles AWS IoT TLS publish/subscribe and shadow sync
  • Fetches mission artifacts via HTTPS from S3 URLs
  • Drives visual game status over SPI display chain
  • Exchanges deterministic frames with UNO over UART

UNO Co-processor

  • Stepper control through ULN2003 driver
  • Ultrasonic scan timing and sector reduction
  • Servo actuation and buzzer signal timing
  • Low-level sampling for DHT11, photoresistor, tilt inputs

Sensor Frame (UNO -> CC3200)

$S,<sector>,<distance_cm>,<lux>,<tilt>,<temp_c>,<hum_pct>\n

Control Frame (CC3200 -> UNO)

$C,<stepper_deg>,<servo_deg>,<buzzer_mode>,<rgb_state>\n

Communication Protocols

  • UART: CC3200 to UNO command/data stream
  • SPI: CC3200 to display/indicator pipeline using 74HC595
  • I2C: expansion sensor/auxiliary bus support
  • HTTPS/MQTT over TLS: cloud mission and telemetry paths

Sensor Set

  • HC-SR04 ultrasonic sensor
  • Photoresistor (ambient anomaly sensing)
  • Tilt ball switch (tamper event)
  • DHT11 (thermal context)
  • IR receiver (human control channel)

AWS IoT Core Role

Shadow stores mission state, round status, and current score. Topic streams carry time-series telemetry for live cloud judging.

Lambda Role

Lambda functions generate mission scripts, compute weighted scoring, and build replay manifests.

S3 Role

S3 stores versioned mission JSON, replay artifacts, and leaderboard snapshots used by the project webpage and final report evidence.

3. Implementation Goals

Minimal Goals

  • Standalone boot sequence with no host control terminal
  • Two-player local interaction using joystick + IR controls
  • Stepper radar scan and servo shield actuation
  • UART frame exchange between master and coprocessor
  • AWS IoT telemetry publish with mission request handshake

Target Goals

  • Cloud mission generation by Lambda and mission retrieval from S3
  • Multi-sensor fusion for sector threat scoring
  • On-device live score output on LCD + 4-digit display
  • Cloud judging and winner computation per round
  • Replay artifact persisted to S3 after each run

Stretch Goals

  • Adaptive mission difficulty from prior run telemetry
  • Multiple arena profiles selectable from cloud dashboard
  • Anomaly classifier to weight threat confidence per sector
  • Automatic leaderboard panel on project webpage

Verification Metrics

  • Mission fetch success from cloud in at least 9/10 runs
  • Round state transitions without deadlock in at least 10/10 runs
  • Telemetry loss under 1% over a two-minute match
  • Match start latency under 3 seconds after mission load
  • Replay artifact generated for every demonstration round
